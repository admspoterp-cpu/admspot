Push notifications iOS + Firebase (FCM igual ao Android)
============================================================

1) Firebase Console (https://console.firebase.google.com) — projeto "admspot-finance"
   - Adicionar app iOS com Bundle ID: com.admspot.finance (igual ao capacitor.config / Xcode)
   - Descarregar GoogleService-Info.plist
   - Colocar em: ios/App/App/GoogleService-Info.plist
   - No Xcode: arrastar o ficheiro para o grupo "App", marcar target "App" (Copy Bundle Resources)
   - Alternativa: copiar o plist descarregado por cima do exemplo e fazer commit (equipa com cuidado com chaves em repositório)

2) Apple Developer — chave APNs para o Firebase
   - Keys > criar key com "Apple Push Notifications service (APNs)"
   - Fazer download do .p8
   - Firebase > Project settings > Cloud Messaging > Apple app configuration
   - Carregar a chave APNs (.p8), Team ID e Key ID

3) Appflow / build
   - O perfil de provisionamento usado no build tem de incluir a capability "Push Notifications"
   - O ficheiro GoogleService-Info.plist tem de existir no repositório (ou ser injetado no pipeline) antes do build nativo
   - Depois de alterar Podfile: na macOS, em ios/App: pod install

4) Token FCM
   - Com GoogleService-Info.plist e FirebaseMessaging, o token enviado ao JS é o token FCM (mesmo endpoint /api/central/v1/push-fcm-token que no Android)
   - Sem o plist, o app usa apenas o token APNs em hex — o backend teria de tratar plataforma diferente

5) aps-environment em App.entitlements
   - "production" — adequado a IPA/App Store e TestFlight (Appflow)
   - Para testes locais com perfil de desenvolvimento, pode ser necessário "development" no Xcode
     (se ficar "production" com certificado de desenvolvimento, o registo APNs falha e não há push)

6) Token FCM no nativo (AppDelegate)
   - O token deve ser enviado ao JS via MessagingDelegate `messaging(_:didReceiveRegistrationToken:)`.
     Usar só `Messaging.messaging().token { }` logo após `apnsToken = deviceToken` é pouco fiável (token muitas vezes nil).

7) Backend / payload FCM
   - O mesmo envio que só preenche `notification`/`data` para Android pode não entregar no iOS.
   - Use a API FCM v1 com bloco `apns` / `fcm_options` para iOS, ou envie notificação multi-plataforma no console Firebase.
   - Em iOS, alertas visíveis precisam de payload APNs (ex.: `apns.payload.aps.alert` ou `mutable-content` conforme o caso).
   - Confirme no gestor que o token guardado para o utilizador é o token FCM longo (não só hex APNs de 64 caracteres).

8) Appflow + TestFlight (sem Xcode local)
   - Repositório: `GoogleService-Info.plist` no target (Copy Bundle Resources), `App.entitlements` com Push, `Podfile` com FirebaseMessaging — já no projeto.
   - Certificados Appflow: perfil de distribuição iOS com capability "Push Notifications" para o App ID com.admspot.finance.
   - Firebase Console > Cloud Messaging: chave APNs .p8 carregada (a mesma chave serve development e production na Firebase).
   - TestFlight usa ambiente APNs production — `aps-environment` = production no repo está correto para IPA da loja/Appflow.
   - Se o push falhar só no iOS mas o Android funciona: quase sempre (a) payload FCM sem secção `apns`, ou (b) token iOS não registado na API, ou (c) APNs não ligado ao projeto Firebase.

9) AppDelegate (token → Capacitor)
   - O token é reenviado duas vezes (imediatos + ~1,2s) para o mesmo `NotificationCenter` para contornar o plugin JS ainda não ter o observer ativo.
   - `application(_:didReceiveRemoteNotification:fetchCompletionHandler:)` encaminha para `Messaging` (mensagens em background).
